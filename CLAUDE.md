# enot - an earthy color scheme robust to dichromacy

The enot theme (vim, wezterm, mc, ranger, site) plus six base palette
variants for comparison. All colors are generated by the optimizer;
never hand-pick colors or edit generated artifacts by hand.

## Pipeline

- palette.py -> palettes.json, preview.html; the enot variant is assembled from enot.json;
- optimize.py -> enot.json: 7 accents + 16 ANSI per theme, max-min pairwise CIEDE2000 across normal vision, protanopia and deuteranopia, then an aesthetic pass toward gruvbox anchors;
- resolve.py -> colors.json: the color specification - every role at three depths (hex, xterm-256, ANSI slot); 256 is solved by the same solve over candidates from the xterm grid with truecolor anchors (naive quantization merges roles down to dE00 0); the single source of colors for renderers;
- check.py - the single regression: metric thresholds of colors.json per depth + every color literal in artifacts belongs to the specification; cvd.py -> cvd.html (report only);
- renderers (thin substitution from colors.json, no color math inside): vim.py -> vim/colors/enot.vim; wezterm.py -> wezterm/*.toml; mc.py -> mc/*.ini (dark/light x truecolor-16M and 256; without truecolor mc falls back to default entirely, so the 256 variant is mandatory); ranger.py -> ranger/colorschemes/enot.py (ANSI indices 0-15 on top of the terminal palette, one file for both themes, ranger >= 1.9.3);
- site.py -> site/ (invariants and metrics on index come from colors.json and check.CHECKS);
- make all - full cycle (check runs last and validates artifacts); make deploy - site rebuild + commit + push.

## Invariants

- changing colors = changing constraints or anchors in optimize.py (truecolor) or tolerances in resolve.py (256), then recompute; after any change run make all, check must pass;
- dichromacy models: Vienot 1999 (protan/deutan), Brettel 1997 (tritan); Machado 2009 is only valid for anomalous trichromacy;
- tritanopia is not part of hard constraints (project decision), report only;
- contrast of text roles to background >= 4.5:1 at all depths; distance from fg/comment >= dE00 10 (bright ANSI and 256 accents - 6.5, stricter is physically unattainable); red C* >= 22 (21.9 after hex roundtrip, threshold 21.5);
- ANSI: normal and bright lightness tiers do not overlap; in the light theme bright is darker than normal (like gruvbox light);
- 256 depth: dark accents >= 7.0, light >= 2.0 - the physical limit of the xterm grid at 4.5:1 contrast (the aqua-blue pair); the full guarantee holds only in truecolor;
- vim scheme attributes are always explicit (gui=NONE), otherwise StatusLine/IncSearch inherit reverse.

## Site

- site/ - a symlink to the sibling checkout of the separate git repository git@github.com:enot-theme/site.git, GitHub Pages from main, https://enot-theme.github.io/site/ ;
- site/ is build output: edit only site.py; llms.txt and llms-full.txt are generated too;
- wezterm install snippets - config_builder style, no top-level return in examples.

## Language and commits

- the organization language and the language of all artifacts without exception is English: sources, comments, palette names, docs, site, commits;
- commits: semantic, one line, up to 80 chars, English, DCO (git commit -s), no Co-Authored-By.

## Documentation

- docs/enot.en.md is the single method document; when the method or numbers change, update it and llms-full (by rebuilding the site);
- key numbers: min dE00 8.3 (accents) and 7.2 (ANSI) in truecolor, 7.3/2.2 (256 accents, dark/light); the site takes numbers from colors.json automatically; in write-ups verify against metrics.
