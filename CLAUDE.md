# enot - an earthy color scheme robust to dichromacy

The enot theme (vim, wezterm, mc, ranger, site). All colors are
generated by the optimizer; never hand-pick colors or edit generated
artifacts by hand. The exploratory palette variants and HTML previews
were removed in 2026-07; they live in git history.

## Pipeline

- palette.py - no artifacts: shared constants (SLUG, ROLES, ANSI_ORDER), LCh -> sRGB math, the flint base and enot_variant() assembling the theme from enot.json;
- optimize.py -> enot.json: 7 accents + 16 ANSI per theme, max-min pairwise CIEDE2000 across normal vision, protanopia and deuteranopia, then an aesthetic pass toward gruvbox anchors;
- resolve.py -> colors.json: the color specification - every role at three depths (hex, xterm-256, ANSI slot); 256 is solved by the same solve over candidates from the xterm grid with truecolor anchors (naive quantization merges roles down to dE00 0); the single source of colors for renderers;
- check.py - the single regression: metric thresholds of colors.json per depth + artifact conformance derived from ports/*/port.json ("16M" in depth -> every hex belongs to the spec, c256_pattern -> extracted 256 indexes belong to the spec, ansi-only depth -> no hex at all), so a new port is checked without touching check.py; cvd.py holds the simulation matrices and dE00 only;
- build.py - the port engine: renders every port under ports/ from colors.json (no color math), writes artifacts to their install paths and ports/registry.json (the coverage manifest). A port is ports/<app>/ with a port.json manifest of one kind: template (a ${var} file rendered per theme via string.Template, no Python - the path for the long tail of simple key=value formats, e.g. wezterm -> wezterm/*.toml); module (a render.py exposing render(spec) -> {path: text}, for formats whose emission needs logic - vim -> vim/colors/enot.vim, mc -> mc/*.ini with dark/light x truecolor-16M and 256, since mc without truecolor falls back to default entirely so the 256 variant is mandatory); static (a file copied verbatim - ranger -> ranger/colorschemes/enot.py, ANSI indices 0-15 on the terminal palette, one file for both themes, ranger >= 1.9.3). A new simple app is one template + one manifest, no Python;
- assets.py -> docs/assets/*.svg: the README design reference rendered from colors.json - editor samples and palette sheets per theme (hex-checked via assets.HEX_CHECKED) and cvd.svg, the gruvbox-vs-enot strip under simulated dichromacy with measured minima (exempt from the hex check by design: foreign palette and simulation outputs);
- sitedata.py -> build/site/ (site.json bundle + llms.txt/llms-full.txt + downloads): the pipeline's contract with the site. All color math stays here; the bundle carries the vision simulations (cvd.SIM_FUNCS run over enot and gruvbox for every theme/vision, so the live switcher is a lookup not a reimplementation), colors.json metrics, the check.py gates, headline numbers and ports/registry.json; the downloads list is derived from the registry (every port output + colors.json). The site never recomputes colors;
- make build - full pipeline (optimize resolve render assets check), the CI gate; make all = build; make sitedata - emit the bundle; make deploy - build + sitedata, then sync the bundle into the Astro site (site/src/data + site/public) and push, whose CI builds and deploys. CI (.github/workflows/ci.yml) runs make build on push/PR and fails on any check violation or when enot.json, colors.json or artifacts are not regenerated from the spec.

## Invariants

- changing colors = changing constraints or anchors in optimize.py (truecolor) or tolerances in resolve.py (256), then recompute; after any change run make all, check must pass;
- dichromacy models: Vienot 1999 (protan/deutan), Brettel 1997 (tritan); Machado 2009 is only valid for anomalous trichromacy;
- tritanopia is not part of hard constraints (project decision); it is simulated only for the site's vision switcher;
- contrast of text roles to background >= 4.5:1 at all depths; distance from fg/comment >= dE00 10 (bright ANSI and 256 accents - 6.5, stricter is physically unattainable); red C* >= 22 (21.9 after hex roundtrip, threshold 21.5);
- ANSI: normal and bright lightness tiers do not overlap; in the light theme bright is darker than normal (like gruvbox light);
- 256 depth: dark accents >= 7.0, light >= 2.0 - the physical limit of the xterm grid at 4.5:1 contrast (the aqua-blue pair); the full guarantee holds only in truecolor;
- vim scheme attributes are always explicit (gui=NONE), otherwise StatusLine/IncSearch inherit reverse.

## Site

- site/ - a symlink to the sibling checkout of git@github.com:enot-theme/enot-theme.github.io.git, an Astro project (not build output); as the organization site it is served at the root: https://enot-theme.github.io/ ;
- Pages source is GitHub Actions (not a branch): the site repo's deploy workflow runs astro check + npm test, then astro build and publishes; no base path, directory-style URLs (/vim/, /apps/);
- the site owns presentation (Astro pages, CSS, install prose in src/data/install.ts); the pipeline owns numbers (src/data/site.json, synced by make deploy). Edit prose/layout in the site repo, never hand-edit src/data/site.json or public/ scheme files;
- /apps/ renders the coverage matrix from the port registry in the bundle; a new port appears there automatically;
- wezterm install snippets - config_builder style, no top-level return in examples;
- analytics: GoatCounter, code in site/src/data/config.ts (register the code to receive data; '' disables).

## Language and commits

- the organization language and the language of all artifacts without exception is English: sources, comments, palette names, docs, site, commits;
- commits: semantic, one line, up to 80 chars, English, DCO (git commit -s), no Co-Authored-By.

## Documentation

- docs/enot.en.md is the single method document; when the method or numbers change, update it, then make sitedata regenerates llms-full.txt and make deploy publishes it;
- key numbers: min dE00 8.2 (accents) and 7.2 (ANSI) in truecolor, 7.3/2.2 (256 accents, dark/light); the site takes numbers from colors.json automatically; in write-ups verify against metrics.
